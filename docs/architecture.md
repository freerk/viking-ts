viking-ts is a monorepo with two packages: a NestJS REST API server and an OpenClaw context-engine plugin.

## NestJS modules

The server is organized into eight modules, all imported by the root `AppModule`:

| Module | Type | Responsibility |
|--------|------|---------------|
| `HealthModule` | Feature | `/health` endpoint |
| `MemoryModule` | Feature | Memory CRUD, search, session capture |
| `ResourceModule` | Feature | Resource CRUD, search |
| `SkillModule` | Feature | Skill CRUD, search |
| `VikingUriModule` | Feature | URI parsing, `ls`, `tree` endpoints |
| `StorageModule` | Global | SQLite metadata store + vectra vector store |
| `EmbeddingModule` | Global | Embedding generation via OpenAI SDK |
| `LlmModule` | Global | LLM completions (abstraction, extraction) |

Global modules (Storage, Embedding, LLM) are instantiated once and injected across all feature modules.

## L0/L1/L2 tiered memory

Every record (memory, resource, skill) stores three levels of abstraction:

### L0: Abstract (~50 tokens)

A single-sentence summary generated by the LLM. This is the text that gets embedded into the vector index for semantic search. When search results are returned, L0 is included as `l0Abstract`.

**Generation prompt**: "Generate a one-sentence abstract of the following content. Maximum 50 tokens."

**Fallback**: first 100 characters of the original text.

### L1: Overview (~500 tokens)

A structured summary with bullet points, generated by the LLM. Provides enough context to decide whether to fetch the full L2 content.

**Generation prompt**: "Generate a concise overview with key points as bullet points. Maximum 500 tokens."

**Fallback**: first 500 characters of the original text.

### L2: Content (full text)

The original untruncated text, stored in SQLite. Retrieved on demand via the `GET /:id` endpoints.

### Why three tiers?

Vector search works best with concise, focused text. Raw content can be long and noisy, degrading search quality. By embedding the L0 abstract instead of the full text, search results are more relevant. L1 provides a middle ground: enough detail for an agent to decide if this memory matters, without loading potentially thousands of tokens of L2 content.

The tiers are generated concurrently via `Promise.all()` on record creation. If the LLM is unavailable (no API key, network error), the system degrades gracefully by slicing the original text.

## Storage layer

### SQLite (metadata store)

Uses better-sqlite3 with WAL mode and foreign keys enabled. Database file: `~/.viking-ts/data/db/viking.db`.

**Tables**:

| Table | Purpose |
|-------|---------|
| `memories` | Memory records with type, category, agentId, all three tiers |
| `resources` | Resource records with title, sourceUrl, all three tiers |
| `skills` | Skill records with name, description, tags (JSON), all three tiers |
| `sessions` | Session metadata (agentId, userId) |
| `session_messages` | Individual messages within sessions (role, content) |

**Indexes**: on `agent_id`, `user_id`, `uri`, `type`, `category`, `name` for query performance.

SQLite handles structured queries: filtering by agentId, category, or type; pagination with limit/offset; and storing the full L2 content.

### vectra (vector store)

Uses the `vectra` library, a pure TypeScript vector database that stores embeddings as JSON files on disk.

**Three independent indexes**:
- `~/.viking-ts/data/vectors/memories/`
- `~/.viking-ts/data/vectors/resources/`
- `~/.viking-ts/data/vectors/skills/`

Each index stores items with:
- `id`: UUID matching the SQLite record
- `vector`: embedding array (e.g. 1536 dimensions for OpenAI `text-embedding-3-small`)
- `metadata`: key-value pairs (uri, text, type, category, etc.)

**Search flow**:
1. Query text is embedded via the EmbeddingService
2. The vector is passed to `index.queryItems(vector, limit)` for KNN search
3. Results are filtered by `scoreThreshold` (minimum similarity)
4. Metadata is mapped to `SearchResult` DTOs

vectra uses cosine similarity for scoring. Results range from 0 (unrelated) to 1 (identical).

## Embedding service

A global singleton that wraps the OpenAI SDK. Configured via environment variables.

The service works with any OpenAI-compatible embedding API:
- **OpenAI**: `text-embedding-3-small` (1536d), `text-embedding-3-large` (3072d)
- **Ollama**: `nomic-embed-text` (768d), `mxbai-embed-large` (1024d)
- **Azure OpenAI**, **LiteLLM**, **vLLM**: any endpoint that accepts `/v1/embeddings`

Key methods:
- `embed(text)`: single text to vector
- `embedBatch(texts)`: multiple texts in one API call

Empty text returns a zero-filled vector of the configured dimension.

## LLM service

A global singleton for text generation, also via the OpenAI SDK. Uses `temperature=0` for deterministic output.

Key methods:
- `generateAbstract(content)`: creates the L0 tier
- `generateOverview(content)`: creates the L1 tier
- `extractMemories(messages)`: analyzes a conversation and returns structured memories with categories
- `complete(system, user, maxTokens)`: raw completion for custom prompts

Memory extraction parses the LLM response as JSON, validates the schema, and returns an array of `{ text, category }` objects. Parse failures return an empty array.

### Two deployment modes

**Local mode**: the plugin spawns the server as a child process via `ProcessManager`. It sets `PORT`, `HOST`, and `STORAGE_PATH` as environment variables, waits for the health check to pass (up to 15s), and connects.

**Remote mode**: the plugin connects to an existing server via `baseUrl`. Useful when running viking-ts as a standalone service.

### agentId resolution

The agentId determines which agent's memories are accessed. Resolution priority:

1. `setAgentId()` override (per-call)
2. `config.agentId` (set at init time)
3. `"default"` (fallback)

### Text sanitization

Before sending messages for auto-capture, the plugin strips:
- `<relevant-memories>` blocks (prevents recursive memory storage)
- `<conversation-metadata>` blocks
- ISO timestamps from line starts
- Excessive whitespace

### Capture filtering

In **semantic** mode (default), all messages longer than 10 chars are captured. In **keyword** mode, messages must match trigger patterns like "remember", "my name is", "I prefer", email addresses, etc.

## Request/response format

All API responses follow a consistent envelope:

```typescript
// Success
{
  "status": "ok",
  "result": T,
  "time": 0.042  // seconds
}

// Error
{
  "status": "error",
  "error": {
    "code": "NOT_FOUND",
    "message": "Memory not found"
  },
  "time": 0.001
}
```

## Validation

Input validation uses `class-validator` with a global `ValidationPipe`:
- `whitelist: true`: strips unknown properties
- `forbidNonWhitelisted: true`: rejects requests with unknown properties
- `transform: true`: auto-transforms query params to correct types

All string inputs have maximum length constraints. Enums are validated against allowed values.

## Data flow: creating a memory

1. Client sends `POST /api/v1/memories` with `{ text, category, agentId? }`
2. `ValidationPipe` validates the DTO
3. `MemoryService.create()` is called
4. UUID is generated, URI is constructed (`viking://user/memories/{category}/{id}`)
5. LLM generates L0 abstract and L1 overview concurrently
6. Record is inserted into SQLite with all three tiers
7. L0 abstract (or text fallback) is embedded via EmbeddingService
8. Embedding + metadata are inserted into vectra
9. Full `MemoryRecord` is returned in the API response

## Data flow: semantic search

1. Client sends `GET /api/v1/memories/search?q=query&limit=6&scoreThreshold=0.01`
2. Query text is embedded via EmbeddingService
3. vectra performs KNN search against the memories index
4. Results below `scoreThreshold` are filtered out
5. Results are mapped to `SearchResult` DTOs (id, uri, text, score, l0Abstract, category, type)
6. Response is returned with timing information

## Data flow: session capture

1. Client sends `POST /api/v1/sessions/capture` with `{ messages, agentId? }`
2. Session record is created in SQLite
3. Messages are stored in `session_messages` table
4. LLM extracts structured memories from the conversation
5. Each extracted memory is created via the standard memory creation flow (L0/L1/L2 + embedding)
6. Response includes count and list of extracted memories
