## What L0/L1/L2 means across all context types

OpenViking uses the same three-level structure for memories, resources, and skills:

| Level | Field | Source | Content |
|---|---|---|---|
| L0 | `abstract` | LLM-generated | One-sentence summary ~100 tokens. Used for quick semantic match. |
| L1 | `overview` | LLM-generated | Structured Markdown ~2000 tokens. Used for planning/navigation. |
| L2 | `content` | Raw file content | Full original text. Read on demand. |

**For memories**: L0/L1/L2 come from `compression/memory_extraction.yaml` output fields.
**For resources**: L0/L1 come from `parsing/context_generation.yaml` (called during resource ingestion).
**For skills**: L1 comes from `skill/overview_generation.yaml`; L0 from description field.

Currently viking-ts:
- Memories: L0 = `content.slice(0,256)` ❌, L1 = `""` ❌, L2 = raw text ✅
- Resources: L0 = `content.slice(0,256)` ❌, L1 = `""` ❌, L2 = raw text ✅
- Skills: L0 = `content.slice(0,256)` ❌, L1 generated by semantic queue but not stored ❌

## Phase A — Memory extraction prompt (CRITICAL)

**Source**: `openviking/prompts/templates/compression/memory_extraction.yaml`

### Task
Replace `memoryExtractionPrompt()` in `packages/server/src/llm/prompts.ts` with a faithful port
of the YAML template. Read the full 401-line YAML file. Copy every instruction, every few-shot
example, every output format rule verbatim into the TypeScript string.

**Key differences from current implementation:**

1. **Output fields**: LLM must return `abstract` (L0), `overview` (L1), `content` (L2) per memory — not just `text`
2. **Per-category format rules** (copy from YAML):
   - `profile`: abstract = `"User basic info: [description]"`, overview uses `## Background Info` heading
   - `preferences`: abstract = `"[Topic]: [Description]"` (merge key format), overview uses `## Preference Domain` / `## Specific Preferences`
   - `entities`: abstract = `"[Entity name]: [description]"`, overview uses `## Basic Info` / `## Core Attributes`
   - `events`: abstract = specific description, overview uses `## Decision Content` / `## Reason` / `## Result`
   - `cases`: abstract = `"[Problem] → [Solution]"`, overview uses `## Problem` / `## Solution`
   - `patterns`: abstract = `"[Pattern name]: [steps]"`, overview uses `## Trigger Condition` / `## Process Flow`
   - `tools`: requires `tool_name` field (REQUIRED), plus `best_for`, `optimal_params`, `common_failures`, `recommendation`
   - `skills`: requires `skill_name` field (REQUIRED), plus `best_for`, `recommended_flow`, `key_dependencies`
3. **Few-shot examples**: Copy ALL examples from the YAML verbatim into the prompt
4. **Output JSON format**: `{ "memories": [{ "category": "...", "abstract": "...", "overview": "...", "content": "...", ...tool/skill fields... }] }`
